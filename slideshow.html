1. 项目背景与设计理念
Why Chaos Fire Drill
🔥 课程痛点
  • 纸上谈兵：学生只会 `kubectl apply`，遇到故障完全不知道从何排查。K8s 概念抽象，缺乏实机排障体验。
  • 可观测性门槛高：云原生系统极其复杂，指标泛滥，初学者难以形成科学的运维直觉。
  • AI 落地浅尝辄止：大模型多用于生成配置代码等"离线"场景，从未与真实生产环境深度穿透形成自愈闭环。
🚒 解决方案：竞速沙箱
  • 混沌工程游戏化：把故障定位与修复抽象为"火灾扑救竞速"，引入时间压力与预算开销约束。
  • 人 🆚 AI 同台博弈：基于 DeepSeek LLM 封装的 AI DevOps Agent 与人类同台诊断，实时对比决策效率。
  • K8s-Native 物理靶场：绝非模拟！每一次 Pod 挂死、服务熔断都作用于真实容器，自愈指令也实机运行。
  • 三实验融合：从 OpenStack IaaS → K8s CaaS → 混沌工程 → AIOps，一条完整的云原生学习链路。

3. 系统架构与核心模块
Architecture & Module Design
⚙️ 后端模块架构
  • StateMachine — 5 状态 FSM (IDLE→INJECTING→DIAGNOSING→SCORING/TIMEOUT)，不可变快照、Listener 模式
  • ChaosInjector — 8 种故障随机注入，按难度分层(Easy 6选1 / Hard 8选2)，每故障自带 restoreFn
  • AiEngine — DeepSeek LLM + 规则引擎双模诊断，流式输出 token-by-token，3-8s 自动修复
  • K8sClient — Mock/Real 双模式 kubectl 客户端，内存态完整模拟 K8s 集群状态
  • Scorer + CostCalculator — SACS 四维评分(速度35+准确20+规范15+成本30=100)
  • TerminalProxy — 白名单(7前缀)+黑名单+命名空间自动注入，防御破坏性命令
4. AI 引擎与混沌故障体系
AI Engine & 8 Chaos Faults
🤖 AI 引擎架构
  • LLM 诊断：基线快照 vs 当前状态 → Prompt 工程 → DeepSeek API (temperature=0.1)
  • 流式输出：AsyncGenerator 逐 token yield，前端实时展示 AI 思考过程
  • 规则引擎回退：无 API Key 时自动切换 — 7 级优先级级联检测
  • 修复验证：attemptRepair() 检查命令模式(scale+replicas / exec+pkill+tc+rm / rollout restart)，拒绝诊断命令
  • 竞速设计：AI 给人类 10s 先手 → 诊断 3-8s → 超时自动修复
规则引擎优先级：notReady Pods → CPU>80% → Memory>80% → restarts>0 → available=0 → ready<expected

5. 创新点总览
Core Innovations
AI 竞速
  • AI 不是助手，是限时对手
  • LLM 流式诊断 token-by-token
  • 超时 AI 自动修复并展示全过程
  • LLM-agnostic：换任何 API 即用
资源监控
  • 实时 CPU/内存使用率柱状条
  • cpu-stress/memory-leak 一眼定位
  • Pod 详情可折叠展开
  • 故障摘要 + 修复命令提示
Webhook
  • 故障注入时弹出 on-call 通知
  • Critical 级别红色+抖动动画
  • 实战模式隐藏原因仅给 URL
  • 15 秒自动消失，可手动关闭
虚拟账单
  • 每轮游戏出模拟云账单
  • Pod费+停机费+命令费+过度配置费
  • 成本权重30%，影响胜负
  • 培养 FinOps 成本意识
安全代理
  • 白名单仅允许 7 种操作
  • 拦截 delete/apply/create 等
  • 自动注入命名空间 -n
  • 防 shell 注入 (;|&&||``|$())
Mock 双模
  • Mock 模式：内存态完整 K8s
  • 零依赖运行，无需真实集群
  • 真实模式：kubectl 直连集群
  • 两种模式前端无差别
K8s-Native
  • 物理集群穿透：每一次状态反馈均直接作用于真实容器节点
  • 轻量级故障注入：基于 Linux 自带 tc 及 dd 指令，资源开销几乎为零
  • 秒级自愈还原：退出或超时自动回滚清理，维持靶场无残留洁净状态
Dogfooding
  • game-server 运行在攻击的集群内
  • ClusterRole 跨命名空间权限
  • 单 Pod 部署(前端+后端+kubectl)
  • Secret 注入 LLM 凭证

Cost 成本
  • Speed：人与 AI 时间比分配。人用时/总用时越低，分越高
  • Accuracy：命令模式匹配故障类型。8 级精度，精准修复满分
  • Standard：kubectl 语法规范。AI 默认满分(+15)，人类需争夺
  • Cost：谁的云账单更低。关键决胜维度，占比 30%
💸 云成本模拟 (FinOps)
收费项费率
Pod 运行费$0.03 /pod/分钟
停机损失费$0.05 /秒
命令执行费$0.02 /条
过度配置罚款$0.15 /多余pod/分钟

设计意图：快速盲目 scale 到高副本 = 过度配置罚款。慢速精确修复 = 停机费高。最优策略：快速 + 精准 + 经济。

8. 总结与未来展望
Summary & Future
🌟 收获
  • 面向故障的设计思维：通过混沌对抗，深刻理解健康检查探针、重启退避对服务韧性的保障价值。
  • AIOps 自愈可行性：LLM + 规则引擎在真实 K8s 集群中实现 OODA 闭环(Observe-Orient-Decide-Act)。
  • 安全 Agent 设计：白名单+黑名单+命名空间隔离=防御深度架构。
🚀 未来方向
  • 多人竞速模式：双人同集群排错，实时指令对比
  • 故障回放系统：录制+回放操作序列，用于教学复盘
  • 双 Agent 对抗："破坏者Agent" vs "自愈Agent" 左右互搏
💡 核心贡献
  • 教育创新：游戏化混沌工程 + 竞争驱动学习
  • 技术融合：IaaS→CaaS→AIOps 全栈覆盖
  • 工程实践：108 测试 + Docker 部署 + RBAC 安全
  • AI 创新：LLM 竞速对手 + 流式诊断 + 自动修复
